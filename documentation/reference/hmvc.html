umi\hmvc

Введение

Компонент umi\hmvc предоставляет API для создания иерархических приложений с Model View Controller архитектурой. Приложение строится из иерархически составленных компонентов, каждый из которых представляет из себя тройку MVC.

Model-View-Controller (MVC)

Рассмотрим определение классической схемы MVC:

Figure 1. Model-View-Controller


Модель(Model) предоставляет знания: данные и методы работы с этими данными, реагирует на запросы, изменяя своё состояние. Не содержит информации, как эти знания можно визуализировать. В моделях содержится вся бизнес-логика приложения.

Контроллер(Controller) обеспечивает связь между пользователем и системой: контролирует ввод данных пользователем и использует модель и представление для реализации необходимой реакции. Контроллер является тонкой частью приложения, в нем содержится только код доступа к моделям и передачи результата в слой отображения.

Отображение(View) отвечает за отображение информации (визуализацию). В роли слоя для отображения выступает шаблонизатор.

Задача MVC приложения заключается в выборе необходимого контроллера в зависимости от запроса пользователя. Решение этой задачи обеспечивается благодаря маршрутизации(routing). Подробнее о маршрутизации смотрите компонент umi\route.

Иерархическое model–view–controller приложение

В иерархическом приложении MVC тройка называется Компонентом приложения. Приложение представляет из себя иерархию компонент:

Figure 2. Hierarchical Model-View-Controller


Задача маршрутизатора, в таком случае, не только выбор контроллера, но и выбор компонента. Таким образом, стрелки на изображении показывают все возможные маршруты запроса в приложении.

Использование

Контроллеры

В umi\hmvc существует несколько типов контроллеров.

umi\hmvc\controller\type\BaseController является базовым контроллером. Он предоставляет методы для создания результата работы:

umi\hmvc\controller\type\BaseController::createPlainResponse

Возвращает простой HTTP ответ компонента.

umi\hmvc\controller\type\BaseController::createDisplayResponse

Возвращает отображаемый HTTP ответ компонента. Для создания ответа такого типа, необходимо указать шаблон и параметры шаблона.

umi\hmvc\controller\type\BaseController::createRedirectResponse

Возвращает HTTP ответ компонента с установленным кодом и заголовками переадресации.

Контроллер вывода статических страниц umi\hmvc\controller\type\StaticPageController. В контроллере заданного типа должен быть указано свойство umi\hmvc\controller\type\StaticPageController::$template- шаблон вывода статической страницы.

Модели

Как уже было сказано выше, вся бизнес логика должна храниться в моделях. Соответственно, основной частью любого приложения или компонента можно считать модели. UMI.Framework не накладывает ограничений на реализацию моделей, т.е. моделью с точки зрения UMI.Framework является любой класс.

Когда наша модель зарегистрирована, ее экземпляр можно получить в контроллере следующими способами:

  • Используя конструктор контроллера. Модель принимается как обязательная зависимость. Модель, в таком случае, должна реализовывать интерфейс umi\hmvc\model\IModel.

  • Используя интерфейс umi\hmvc\model\IModelAware и трейт umi\hmvc\model\TModelAware контроллер может получить возможность создавать модели по имени класса или по имени.

Важно

Инъекция моделей, как и их явное создание, возможно только с моделями, зарегистрированными в данном MVC компоненте.

Отображение

Большинству web приложений необходимо отображать информацию. Для этой цели служит слой отображения, который умеет выводить данные с использованием шаблонизатора. Для этой цели umi\hmvc использует компонент umi\templating.

Выбор шаблонизатора делается на уровне одного компонента. Соответственно, приложение может использовать несколько шаблонизаторов одновременно.

Для шаблонизаторов созданы "помощники вида"(View Helper). Они позволяют упростить стандартный набор действий, либо добавить удобный функционал в шаблонизатор. К примеру, в системе есть помощники вида для перевода текста, вывода элементов формы, коллекционирования мета информации и т.п.

Помощники вида

Помощник вида - это помощник шаблона, в котором могут быть использованы зависимости от контекста. В помощниках вида конкретного компонента могут использоваться также зависимости от моделей.

В компоненте umi\hmvc присутствуют стандартные помощники вида. Рассмотрим их список:

getUrl($name = null, array $params = [], $useRequestParams = false)

Помощник для генерации ссылок внутри компонента. Генерирует ссылку с именем $name, с помощью параметров $params. Если флаг $useRequestParams установлен, то параметры, полученные при маршрутизации, будут подставлены в параметры для генерации ссылки.

Компоненты

Компонент в umi\hmvc представляет из себя объект, построенный на базе конфигурации. Она описывает все контроллеры и модели, логику маршрутизации и слой отображения.

Подсказка

Переопределив значение параметра componentClass можно заставить umi\hmvc создавать собственный класс компонента.

Пример конфигурации компонента:

<?php
namespace application\components\example;

use umi\hmvc\component\IComponent;
use umi\route\IRouteFactory;

return [
    IComponent::OPTION_CONTROLLERS => [
        'login'  => __NAMESPACE__ . '\controller\IndexController',
    ],
    IComponent::OPTION_VIEW        => [
        'type'      => 'php',
        'directory' => __DIR__ . '/view',
        'extension' => 'phtml',
    ],
    IComponent::OPTION_ROUTES      => [
        'index'  => [
            'type'     => IRouteFactory::ROUTE_FIXED,
            'defaults' => [
                'controller' => 'index'
            ]
        ],
    ],
];

Для того, чтобы определить вызываемый контроллер или компонент для маршрута необходимо определить переменную маршрута с заданным именем:

IComponent::MATCH_COMPONENT

Переменная маршрута, определяющая необходимый дочерний компонент, которому будет передано управления.

IComponent::MATCH_CONTROLLER

Переменная маршрута, определяющая необходимый контроллер текущего компонента.

Расширенное использование

Контекст

В контроллеры, а также помощники вида компонента, может быть внедрен контекст.

umi\hmvc\context\IContext - это набор контекстно-зависимых сущностей. Контекст компонента представляет из себя: сам компонент, HTTP запрос компонента, результат работы маршрутизатора (опционально).

Контекст компонента можно получить, используя интерфейс umi\hmvc\context\IContextAware. К примеру: 

class Controller extends BaseController implements IContextAware
{
    use TContextAware;
}

В контроллере, а также помощники вида компонента, с внедренным контекстом доступны следующие методы:

TContextAware::hasContext

Проверяет, установлен ли контекст в контроллер.

TContextAware::getContext()

Возвращает контекст, установленный в контроллер.
