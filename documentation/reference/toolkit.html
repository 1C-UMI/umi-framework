umi\toolkit

Введение

UMI.Framework придерживается сервис-ориентированной архитектуры (SOA) и предоставляет набор слабо связанных сервисов, имеющих четко определенные интерфейсы (контракты). Интерфейсы сервисов инкапсулируют детали реализации, что обеспечивает слабую связанность между сервисами и способствует масштабируемости и управляемости создаваемых на фреймворке приложений.

Все реализации сервисов придерживаются принципа инверсии зависимостей (Dependency inversion principle) и объявляют об используемых зависимостях явно: через конструктор (Constructor injection), через setter (Setter injection), либо через интерфейс (Interface injection).

Компонент umi\toolkit представляет собой простой контейнер сервисов (IoC контейнер). Toolkit помогает создавать и конфигурировать сервисы, разрешает зависимости между ними.

Для разрешения зависимостей umi\toolkit\IToolkit поддерживает регистрацию следующих видов сущностей:

Service (сервис) - объект определенного интерфейса, реализующий строго определенную задачу;

Aware interface - интерфейс, который определяет метод для внедрения в объект внешней зависимости;

Toolbox (набор инструментов) - контейнер, готовый к повторному использованию, предоставляющий набор сервисов и aware interfaceces.

Каждый компонент UMI.Framework предоставляет сервисы и интерфейсы через Toolbox, готовый к регистрации и конфигурированию.

Если вы используете за основу своего приложения Skeleton Application, то для использования сервисов желаемого компонента, достаточно подключить конфигурацию его toolbox в настройках загрузчика приложения configuration/bootstrap.php. Там же можно задать настройки инструментов. Более подробно о настройках и подключении описано в документации по соответствующему компоненту.

Регистрация сервисов

Любой интерфейс может стать сервисом, для этого необходимо зарегистрировать в Toolkit функцию , которая будет создавать и конфигурировать конкретную реализацию данного интерфейса (далее builder).

Простой пример регистрации сервисов

Пусть у нас есть интерфейсы InterfaceA и InterfaceB, а так же их реализации ConcreteA и ConcreteB. Так же есть интерфейс InterfaceC, который имеет две реализации: DefaultC (реализация по умолчанию) и AnotherC. DefaultC имеет зависимость от InterfaceA, а AnotherC от InterfaceB, эти зависимости явно задаются в конструкторе каждого класса, следуя принципу инверсии зависимостей.

interface InterfaceA
{
    public function methodA();
}

interface InterfaceB
{
    public function methodB();
}

interface InterfaceC
{
    public function methodC();
}

class ConcreteA implements InterfaceA
{
    public function methodA()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '().';
    }
}

class ConcreteB implements InterfaceB
{
    public function methodB()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '().';
    }
}

class DefaultC implements InterfaceC
{
    protected $a;

    public function __construct(InterfaceA $a) {
        $this->a = $a;
    }

    public function methodC()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '(). ' . $this->a->methodA();
    }
}

class AnotherC implements InterfaceC
{
    protected $b;

    public function __construct(InterfaceB $b) {
        $this->b = $b;
    }

    public function methodC()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '(). ' . $this->b->methodB();
    }

}

Зарегистрируем интерфейсы InterfaceA, InterfaceB и InterfaceC в качестве сервисов Toolkit:

<?php

use umi\toolkit\IToolkit;
use umi\toolkit\Toolkit;

require __DIR__ . '/../__init__.php';

$toolkit = new Toolkit();

$toolkit->registerService('InterfaceA', function() {
   return new ConcreteA();
});

$toolkit->registerService('InterfaceB', function() {
    return new ConcreteB();
});

$toolkit->registerService('InterfaceC', function($concreteClassName, IToolkit $toolkit) {
    $concreteClassName = $concreteClassName ?: 'DefaultC';

    switch ($concreteClassName) {
        case 'AnotherC' :
            /** @var InterfaceB $serviceB */
            $serviceB = $toolkit->getService('InterfaceB');
            return new AnotherC($serviceB);
        default:
            /** @var InterfaceA $serviceA */
            $serviceA = $toolkit->getService('InterfaceA');
            return new DefaultC($serviceA);
    }
});

/** @var InterfaceA $a */
$a = $toolkit->getService('InterfaceA');
/** @var InterfaceB $b */
$b = $toolkit->getService('InterfaceB');
/** @var InterfaceC $c1*/
$c1 = $toolkit->getService('InterfaceC');
/** @var InterfaceC $c2*/
$c2 = $toolkit->getService('InterfaceC', 'AnotherC');

echo($a->methodA() . PHP_EOL);
// output: Hello world from ConcreteA::methodA().

echo($b->methodB() . PHP_EOL);
// output: Hello world from ConcreteB::methodB().

echo($c1->methodC() . PHP_EOL);
// output: Hello world from DefaultC::methodC(). Hello world from ConcreteA::methodA().

echo($c2->methodC() . PHP_EOL);
// output: Hello world from AnotherC::methodC(). Hello world from ConcreteB::methodB().

В рассмотренном примере, зависимости для каждой из реализаций сервиса InterfaceC разрешаются явным образом. В builder сервиса доступно имя класса желаемой реализации сервиса, а так же Toolkit, который используется в качестве поставщика сервисов (Service Locator).

Создание объектов с использованием прототипа (umi\toolkit\prototype\IPrototype)

Ранее мы зарегистрировали сервисы InterfaceA, InterfaceB и InterfaceC. Builder сервиса InterfaceC создает конкретные экземпляры сервиса и разрешает зависимости для каждой реализации явным образом, то есть билдер знает все о каждой реализации сервиса. При добавлении новой реализации сервиса, необходимо добавить эту реализацию в билдер, при добавлении новой зависимости у конкретной реализации, нам необходимо изменить код билдера. Если проводится постоянный рефакторинг, а так же добавляются все больше сервисов и конкретных реализаций, поддержка билдеров и "ручное" разрешение зависимостей может стать проблемой.

Toolkit умеет автоматически разрешать зависимости от сервисов, которые в нем зарегистрированы. Для автоматического разрешения зависимостей, необходимо создавать конкретные реализации через прототип (umi\toolkit\prototype\IPrototype).

Прототип вычисляет зависимости конкретного класса и умеет автоматически "подставлять" известные Toolkit сервисы в аргументы конструктора. Классу, объекты которого создаются с использованием прототипа необходимо всего лишь объявить в конструкторе необходимые сервисы.

Изменим билдер сервиса InterfaceC таким образом, чтобы он не зависел от конкретных реализаций сервиса:

$toolkit->registerService('InterfaceC', function($concreteClassName, IToolkit $toolkit) {
    $concreteClassName = $concreteClassName ?: 'DefaultC';

    return $toolkit->getPrototype($concreteClassName, ['InterfaceC'])->createInstance();
});

Теперь мы можем добавлять сколько угодно реализаций сервиса без изменения билдера.

Регистрация Aware Interface

Внедрение сервисов с помощью конструктора (Constructor injection) не всегда удобно. Недостаток внедрения с помощью конструктора состоит в том, что конструктор становится сильно перегруженным, так как зависимостей может быть много, некоторые из них могут быть не обязательными (логгер, кэш). При добавлении еще одной зависимости часто приходится пересматривать последовательность аргументов, что сильно усложняет расширение компонентов.

Рекомендуем использовать constructor injection в случаях, когда это обязательные зависимости для всех наследников и их не много, а так же в случаях, когда наследование не планируется совсем (например, в контроллерах MVC-приложения).

Указанные выше проблемы решает другой способ внедрения сервисов - внедрение c помощью интерфейсов (Interface injection). Для каждого сервиса задается интрерфейс (далее aware interface), который определяет метод для внедрения сервиса. Зависимый от сервисов объект должен реализовать такой интерфейс, сохранив у себя ссылку на зависимость.

В разделе "Регистрация сервисов" мы зарегистрировали сервисы InterfaceA и InterfaceB, а так же рассмотрели возможность внедрения этих сервисов с помощью Constructor injection в реализации сервиса InterfaceC. Рассмотрим, как такую же задачу можно решить с помощью Interface injection. Создадим для этих сервисов aware interfaces и зарегистрируем для них в Toolkit функции (injectors), которые будет использовать прототип для внедрения соответсвующих сервисов в объекты:

interface InterfaceAAware {
    /**
     * Устанавливает реализацию сервиса InterfaceA.
     * @param InterfaceA $a
     */
    public function setA(InterfaceA $a);
}


interface InterfaceBAware {
    /**
     * Устанавливает реализацию сервиса InterfaceB.
     * @param InterfaceB $b
     */
    public function setB(InterfaceB $b);
}


$toolkit->registerAwareInterface('InterfaceAAware', function($object, IToolkit $toolkit) {
    if ($object instanceof InterfaceAAware) {
        $object->setA($toolkit->getService('InterfaceA'));
    }
});

$toolkit->registerAwareInterface('InterfaceBAware', function($object, IToolkit $toolkit) {
    if ($object instanceof InterfaceBAware) {
        $object->setB($toolkit->getService('InterfaceB'));
    }
});

Теперь мы можем "освободить" конструкторы реализаций сервиса InterfaceC:

class DefaultC implements InterfaceC, InterfaceAAware
{
    /** @var InterfaceA $a */
    private $a;

    public function setA(InterfaceA $a) {
        $this->a = $a;
    }

    public function methodC()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '(). ' . $this->getA()->methodA();
    }

    protected function getA() {
        if (!$this->a) {
            throw new RuntimeException('Service "InterfaceA" is not injected.');
        }
        return $this->a;
    }
}

class AnotherC implements InterfaceC, InterfaceBAware
{
    /** @var InterfaceB $b */
    private $b;

    public function setB(InterfaceB $b)
    {
        $this->b = $b;
    }

    public function methodC()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '(). ' . $this->getB()->methodB();
    }

    protected function getB() {
        if (!$this->b) {
            throw new RuntimeException('Service "InterfaceB" is not injected.');
        }
        return $this->b;
    }
}

Так как реализации сервиса InterfaceC создаются через прототип, разрешение зависимостей по интерфейсам происходит автоматически. Если нам понадобится добавить еще одну зависимость в какую-либо реализацию, нам необходимо лишь реализовать соответсвующий aware interface.

Для того, чтобы избежать дублирование кода, связанного с внедрением сервиса и не нарушать принцип DRY, можно определить трейт для внедрения сервиса. Использование трейтов позволяет сделать код более читаемым, особенно хорошо это заметно, когда у реализации много зависимостей. Часто бывает удобно сделать в трейте методы быстрого доступа, которые будут передавать управдение соответсвующему сервису (делегирование).

Определим трейт, для внедрения сервиса InterfaceA:

trait TInterfaceAAware {
    /** @var InterfaceA $a */
    private $_a;

    public function setA(InterfaceA $a) {
        $this->a = $a;
    }

    protected function getA() {
        if (!$this->a) {
            throw new RuntimeException('Service "InterfaceA" is not injected.');
        }
        return $this->a;
    }

    protected function methodA() {
        return $this->getA()->methodA();
    }
}

Теперь мы можем использовать этот трейт в реализации сервиса:

class DefaultC implements InterfaceC, InterfaceAAware
{
    use TInterfaceAAware;

    public function methodC()
    {
        return 'Hello world from ' . __CLASS__ . '::' . __METHOD__ . '(). ' . $this->methodA();
    }

}

Хорошо видно, как упростился код и повысилась его читаемость. Если классу потребуются еще зависимости, то достаточно реализовать соответствующий aware interface используя для этого соответствующий трейт.

Компоненты UMI.Framework предоставляют интерфейсы и трейты для внедрения своих сервисов (Пример: umi\cache\ICacheAware, umi\cache\TCacheAware). Если реализации ваших сервисов создаются с использованием прототипа, вы можете использовать их, так как Toolkit автоматически внедряет сервисы по известным ему интерфейсам, выполняя необходимые функции-инжекторы (injectors).

Важно

Не рекомендуется писать в трейтах какую-либо сложную логику, используйте трейты только для внедрения зависимости и для делегирования.

Для избежания конфликтов имен свойств в трейтах с использующими их классами, рекомендуем использовать префикс "_" для приватных и защищенных свойств трейта.

Избегайте, по возможности, использования в трейтах других трейтов, это усложняет логику и приводит к конфликтам. Если трейту необходим какой-либо метод другого трейта, рекомендуем объявить его абстрактным. Использующий трейт класс сможет реализовать этот метод самостоятельно, либо подключить необходимый трейт.

Использование Toolkit в качестве Service Locator

Важно

Мы крайне не рекомендуем использовать Toolkit в качестве Service Locator в приложениях на UMI.Framework, так как это считается плохой практикой. Все коробочные компоненты фреймворка хорошо интегрированы с Toolkit и создают большинство объектов через прототип, что позволяет явно объявлять необходимые зависимости в конструкторе классов или через aware interface, Toolkit внедрит зависимости автоматически.

Если вы все же хотите использовать Toolkit, в качестве Service Locator, то такая возможность поддерживается. Для этого необходимо либо объявить umi\toolkit\IToolkit в параметрах конструктора, либо реализовать интерфейс umi\toolkit\IToolkitAware с помощью трейта umi\toolkit\TToolkitAware.

<?php

use umi\toolkit\IToolkitAware;
use umi\toolkit\TToolkitAware;

class ToolkitAwareService implements IToolkitAware
{
    use TToolkitAware;

    public function fooBar()
    {
        // Теперь нам доступны методы для получения любого сервиса из Toolkit
        $myService = $this->getService('InterfaceA');
    }
}
