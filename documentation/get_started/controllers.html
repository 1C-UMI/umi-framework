Маршрутизация и контроллеры

Каждая страница приложения имеет свой URL, которому соответствует определенный маршрут. Каждый маршрут, в свою очередь, определяет, какой контроллер для обработки запроса пользователя должен быть выполнен.

Главная страница приложения, обрабатываемая основным компонентом приложения, является статической страницей. Для страниц такого типа существует специальный тип контроллеров - контроллеры статических страниц. Для работы таких контроллеров достаточно указать название шаблона. В Skeleton приложении уже существует контроллер главной страницы, являющийся контроллером статической страницы, указывающий на шаблон index, хранящийся в файле /application/view/index.phtml.

Для остальных 4х страниц приложения, относящихся к компоненту блога, нам необходимо сконфигурировать соответствующие маршруты и создать контроллеры.

Регистрация контроллеров

Классы контроллеров блога будем хранить в ранее созданной папке /application/components/blog/controller. Всего нам понадобится 4 контроллера, нам необходимо создать классы контроллеров и задать им уникальные идентификаторы:

Table 1. Контроллеры приложения

ИдентификаторКлассНазначение
post/listapplication\components\blog\controller\IndexControllerКонтроллер вывода списка постов
post/addapplication\components\blog\controller\AddPostControllerКонтроллер добавления поста
post/editapplication\components\blog\controller\EditPostControllerКонтроллер редактирования поста
post/deleteapplication\components\blog\controller\DeletePostControllerКонтроллер удаления поста

UMI.Framework не накладывает ограничений на названия классов контроллеров, однако они обязательно должны реализовывать интерфейс umi\hmvc\controller\IController. Для удобства создания своих контроллеров есть базовый класс контроллера umi\hmvc\controller\type\BaseController и базовый класс контроллера статических страниц umi\hmvc\controller\type\StaticPageController.

Создадим соответствующие классы контроллеров. Чтобы приложение находилось в рабочем состоянии, с помощью метода createPlainResponse() вернем в контроллерах ответ, не требующий шаблонизации. Постепенно мы изменим код контроллеров.

<?php
namespace application\components\blog\controller;

use umi\hmvc\component\request\IComponentRequest;
use umi\hmvc\controller\type\BaseController;

/**
 * Контроллер отображения списка постов блога.
 */
class IndexController extends BaseController
{
    /**
     * Вызывает контроллер. Передает в контроллер HTTP запрос.
     * @param IComponentRequest $request HTTP запрос
     * @return IComponentResponse результат работы контроллера
     */
    public function __invoke(IComponentRequest $request)
    {
        return $this->createPlainResponse("Здесь будет список постов блога");
    }
}

Зарегистрируем контроллеры в конфигурации компонента блога (/application/components/blog/component.config.php), дополнив ее секцией IComponent::OPTION_CONTROLLERS:

namespace application\components\blog;

use umi\hmvc\component\IComponent;
// ...
// зарегистрированные контроллеры компонента
IComponent::OPTION_CONTROLLERS => [
    'post/list' => __NAMESPACE__ . '\controller\IndexController',
    'post/add' => __NAMESPACE__ . '\controller\AddPostController',
    'post/edit' => __NAMESPACE__ . '\controller\EditPostController',
    'post/delete' => __NAMESPACE__ . '\controller\DeletePostController'
],

Регистрация маршрутов

Для того чтобы в определенный момент передать управление от основного компонента его дочерним компонентам, нам необходимо модифицировать настройки маршрутизации в конфигурации основного компонента (файл /application/components/component.config.php,), изменив в секции IComponent::OPTION_ROUTES описание базового маршрута:

namespace application\components\blog;

use umi\hmvc\component\IComponent;
// ...		              
// маршруты текущего компонента
IComponent::OPTION_ROUTES      => [
    // базовый маршрут внутри компонента
    'home' => [
        // тип маршрута
        'type'     => IRouteFactory::ROUTE_EXTENDED,
        // путь(маска) маршрута
        'route'    => '/{lang}',
        // путь(маска) маршрута
        'rules' => [
            'lang' => '[a-z]{2}-[A-Z]{2}'
        ],
        // параметры маршрута по умолчанию
        'defaults' => [
            'lang'       => 'ru-RU',
            'controller' => 'index'
        ],
        // описание вложенных маршрутов
        'subroutes' => [ 
            'component' => [
                // тип маршрута
                'type' => IRouteFactory::ROUTE_SIMPLE,
                // путь(маска) маршрута
                'route' => '/{component}',
                'defaults' => [
                    'controller' => ''
                ],
            ],
        ]
    ]
],
//...

Таким образом, мы добавили новый маршрут home\component, который передает обязанности по определению контроллера своим дочерним компонентам.

Конфигурация маршрутов всегда задается в виде дерева маршрутов, которое маршрутизатор обходит последовательно при сопоставлении запроса пользователя с масками маршрутов. При первом совпадении обход дерева заканчивается, поэтому наиболее часто используемые маршруты выгоднее помещать в начале дерева, а более общие маршруты необходимо помещать в самом конце.

Дополним конфигурацию компонента блога его маршрутами (секция IComponent::OPTION_ROUTES в файле /application/components/blog/component.config.php):

namespace application\components\blog;

use umi\hmvc\component\IComponent;
use umi\route\IRouteFactory;
// ...

// маршруты текущего компонента
IComponent::OPTION_ROUTES => [
    //маршрут добавления поста
    'postAdd' => [
        'type' => IRouteFactory::ROUTE_FIXED,
        'route' => '/add',
        'defaults' => [
            'controller' => 'post/add'
        ]
    ],
    //маршрут редактирования поста
    'postEdit' => [
        'type' => IRouteFactory::ROUTE_SIMPLE,
        'route' => '/edit/{guid:guid}',
        'defaults' => [
            'controller' => 'post/edit'
        ]
    ],
    //маршрут добавления поста
    'postDelete' => [
        'type' => IRouteFactory::ROUTE_SIMPLE,
        'route' => '/delete/{guid:guid}',
        'defaults' => [
            'controller' => 'post/delete'
        ]
    ],
    // базовый маршрут внутри компонента
    'home' => [
        'type' => IRouteFactory::ROUTE_FIXED,
        'defaults' => [
            'controller' => 'post/list'
        ],
    ]
]

Два маршрута postAdd и home это фиксированные маршруты без параметров, т.е. чтобы они были вызваны, адрес в строке браузера должен полностью совпадать с их путем (опция route) . Такие маршруты имеют тип IRouteFactory::ROUTE_FIXED.

Два других маршрута postEdit и postDelete имеют непостоянный последний параметр типа guid. В этом параметре будет передаваться GUID поста блога, который нужно отредактировать или удалить. Маршруты с нефиксированным значением параметров с заданным типом используют тип IRouteFactory::ROUTE_SIMPLE.

Если до этого вы все сделали правильно, то теперь, при запросе URL вида http://имя_вашего_домена/blog и http://имя_вашего_домена/en-US/blog, мы увидим страницу, содержащую текст “Здесь будет список постов блога”. Это говорит о том, что нужный контроллер был вызван и результат его работы был обработан контроллером сетки приложения основного компонента.
