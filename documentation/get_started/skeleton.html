Начало работы

Установка Skeleton приложения

Самым простым способом создания нового приложения является создание приложения на основе Skeleton приложения. Skeleton приложение (скелет приложения) - это простое настроенное MVC приложение, которое уже содержит одну статичную страницу и обработку ошибок.

Таким образом, Skeleton приложение - это рабочее приложение, которое в процессе разработки мы будем расширять.

Исходный код Skeleton приложения лежит на GitHub. Само приложение проще всего установить через Сomposer командой в корневую директорию сервера: 

composer create-project umi/skeleton

После установки у вас появится директория /application с компонентами приложения, директория /configuration с настройками приложения, директория /vendor/umi/framework с UMI.Framework, директория /public_html с публичными файлами, где находится index.php и файл Bootstrap.php, реализующий логику загрузчика приложения.

В результате выполнения описанных действий вы получите работающее приложение на UMI.Framework. Это можно проверить, зайдя на тот домен, для которого вы устанавливали приложение, вы увидете стартовую страничку приложения.

Конфигурация

Настройка приложения начинается с редактирования начальной конфигурации загрузчика приложения, представленной простым php-массивом. Эта конфигурация находится в файле /configuration/bootstrap.php и содержит в себе подключение используемых в приложении наборов инструментов (секция Bootstrap::OPTION_TOOLKIT) и их начальные настройки (секция Bootstrap::OPTION_SETTINGS), необходимые для старта приложения. В Skeleton приложении конфигурация загрузчика содержит минимальный набор используемых наборов инструментов.

return [
/**
* Используемый набор инструментов.
*/
    Bootstrap::OPTION_TOOLKIT => [
        ...
        require($libraryPath . '/i18n/toolbox/config.php'),
        ...
    ],
/**
* Настройки инструментов.
*/
    Bootstrap::OPTION_SETTINGS => [
        ...
        ConfigTools::NAME => [
            'aliases' => [
                '~' => [__DIR__, __DIR__ . '/local'],
                '~/application' => [dirname(__DIR__) . '/application']
            ]
        ],
        ...
    ]
];

Все остальные настройки должны находится в конфигурации приложения в файле /configuration/general.php. Эта конфигурация уже не совсем простой php-массив, а часть масштабной конфигурации, создаваемой с помощью компонента umi\config. Здесь для удобной работы с конфигурацией можно использовать различные управляющие конструкции, предоставляемые umi\config. В Skeleton приложении конфигурация приложения содержит ссылку на ленивую загрузку конфигурации основного MVC компонента. 

return [
    'application' => '{#lazy:~/application/component.config.php}'
];

Обе эти конфигурации мы будем расширять в процессе написания демонстрационного приложения.

MVC компоненты

Приложение UMI.Framework использует модульную архитектуру иерархически связанных MVC компонентов. Приложение состоит из набора таких компонентов. Каждый компонент состоит из набора контроллеров, имеет внутренний маршрутизатор, может регистрировать необходимые модели, отображения, а также дочерние ему компоненты.

Skeleton приложение состоит из одного компонента, который выводит статическую страницу и отвечает за вывод ошибок. Этот компонент мы будем называть основным, он станет родительским компонентом для всех создаваемых компонентов. Класс основного компонента описан в файле /application/Application.php, а его конфигурация в файле /application/component.config.php.

Для демонстрационного приложения мы создадим новый компонент для работы с постами блога.

Маршрутизация и контроллеры

При любом обращении к приложению происходит анализ запроса пользователя, сопоставление этого запроса со списком заданных маршрутов и в итоге вызов соответствующего контроллера. Контроллер связывает модель и представление и на основе запроса пользователя выполняет определенные действия.

Контроллеры и маршруты всегда принадлежат определенному компоненту и описываются в конфигурации этого компонента. Каждый контроллер имеет свой идентификатор, уникальный в рамках компонента. В конфигурации маршрута задается идентификатор контроллера, который будет вызван при совпадении запроса с маршрутом.

Основной компонент содержит два специальных контроллера (контроллер ошибок и контроллер сетки приложения) и контроллер вывода главной страницы:

//...,            
// зарегистрированные контроллеры
IComponent::OPTION_CONTROLLERS => [
    IComponent::ERROR_CONTROLLER   => __NAMESPACE__ . '\controller\ErrorController', //контроллер ошибок
    Application::LAYOUT_CONTROLLER => __NAMESPACE__ . '\controller\LayoutController', //контроллер сетки
    'index'                        => __NAMESPACE__ . '\controller\IndexController', //контроллер главной страницы
], 
//...

Сами контроллеры находятся в директории /application/controller.

Контроллер сетки приложения вызывается всегда, когда контроллер, определенный в результате маршрутизации, успешно отработал, и если результат его работы требует шаблонизации.

Контроллер ошибок вызывается автоматически, когда контроллер, определенный в результате маршрутизации, вызвал исключительную ситуацию (exception).

Для того чтобы вызвать контроллер главной страницы, конфигурация основного компонента содержит следующий маршрут:

//...,            
// маршруты текущего компонента
IComponent::OPTION_ROUTES      => [
    // базовый маршрут внутри компонента
    'home' => [
        // тип маршрута
        'type' => IRouteFactory::ROUTE_EXTENDED,
        // путь(маска) маршрута
        'route' => '/{lang}',
        // путь(маска) маршрута
        'rules' => [
            'lang' => '[a-z]{2}-[A-Z]{2}'
        ],
        // параметры маршрута по умолчанию
        'defaults' => [
            'lang'       => 'ru-RU',
            'controller' => 'index'
        ]
    ]
],
//...

Каждый маршрут имеет имя, тип, маску и параметры маршрута по умолчанию. Имя маршрута необходимо для составления URL из маршрута. Тип маршрута определяет то, как запрос пользователя будет сопоставляться с маской маршрута. Параметры по умолчанию обязательно включают в себя идентификатор контроллера, а также могут определять значения по умолчанию для частей маршрута с переменными значениями. В зависимости от конкретного типа маршрута его конфигурация может иметь дополнительные параметры.

Указанный маршрут home рассчитан на то, что для вывода страниц будет учтен язык, по умолчанию равный русскому. Поэтому маршрут home имеет тип IRouteFactory::ROUTE_EXTENDED и имеет переменную часть lang, значения в которую будут попадать при совпадении с регулярным выражением. Если вы допишете к адресу своего домена параметр языка /en-US, то главная страница будет отображаться на английском языке.

Figure 2. Переведенная главная страница


В результате маршрутизации компонент может либо вызвать свой контроллер, либо передать управление своему дочернему компоненту. После добавления компонента блога мы сконфигурируем основной компонент так, чтобы при определенном маршруте он передавал управление компоненту блога.

Представление и шаблонизация

Шаблоны относятся к определенному компоненту и размещаются в директории, указанной в опции directory секции IComponent::OPTION_VIEW конфигурации компонента.

Каждый контроллер сам указывает, какой шаблон используется для генерации отображения его работы.

Компонент может использовать только один тип шаблонизатора для обработки всех своих шаблонов.

Основной компонент использует php-шаблонизатор, его шаблоны лежат в папке /application/view и имеют расширение phtml:

//...,            
// настройки шаблонизатора
IComponent::OPTION_VIEW        => [
    // тип шаблонизатора
    'type'      => 'php',
    // расширение файлов с шаблонами
    'extension' => 'phtml',
    // путь до шаблонов компонента
    'directory' => __DIR__ . '/view',
],
//...
