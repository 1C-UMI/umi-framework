umi\form

Введение

Компонент umi\form является одним из основных компонентов для работы с данными. Он позволяет предоставить доступ к модификации данных, возможность фильтровать и проверять данные введенные пользователем. Он содержит в себе набор необходимых элементов формы, поддерживает группы полей, коллекции элементов, а также дочерние формы.

Типы сущностей формы

umi\form\IFormEntity - абсолютно любой элемент формы, в том числе и она сама. Это базовый тип для всех остальных типов сущностей формы. Сущности этого типа поддерживают работу с атрибутами и опциями, имеют название, а также имеют метод для проверки корректности своего содержимого.

Поле формы

umi\form\element\IElement - это простой элемент формы, к примеру поле ввода текста или кнопка. Этот элемент не может содержать дочерних элементов.

Группа полей

umi\form\fieldset\IFieldset - группа полей формы, содержит в себе дочерние элементы формы. Это декорирующий элемент, т.е. его использование оправданно только для визуального деления полей, . Группа полей не оказывает влияние на именование элементов внутри себя.

Коллекция полей

umi\form\fieldset\ICollection - коллекция полей, группа полей однотипных элементов. Используется для элементов, количество которых может быть изменено пользователем(к примеру список прикрепленных файлов). При установке значения коллекция автоматически расширяется, т.е. создает необходимое количество элементов внутри себя. Соответственно, в коллекции задается только 1 элемент.

Форма

umi\form\IForm - форма, основной элемент компонента. Форма может быть основной, либо дочерней. Дочерняя форма, в отличии от простой группы полей, изменяет именование своих элементов. Таким образом дочерние формы не могут быть использованы только для декорирования вывода.

Использование

Создание формы

Любая форма, даже самая сложная, задается с помощью конфигурации. Конфигурация описывает все используемые формой элементы в требуемом порядке. Она также описывает атрибуты и опции этих элементов. Приведем пример конфигурации простейшей формы входа на сайт:

return [
    'name'     => 'sampleForm',
    'action'   => '/sample',
    'method'   => 'post',
    'elements' => [
        'login'    => [
            'type'  => Text::TYPE_NAME,
            'label' => 'Login'
        ],
        'password' => [
            'type'  => Password::TYPE_NAME,
            'label' => 'Password'
        ]
    ]
];

Для более простого и понятного описания формы существуют аннотации. Они позволяют сделать некоторые действия с элементом формы (например изменить атрибуты или добавить фильтр).

Аннотация - это элемент конфигурации элемента (например, label), исключая стандартные: type, attributes, options, elements) который описывает параметры формы. К примеру, action и method для формы являются аннотациями.

action

Устанавливает атрибут action у формы. Если объект не является формой, будет брошено исключение.

method

Устанавливает атрибут method у формы. Если объект не является формой, будет брошено исключение.

label

Устанавливает метку(заголовок) элемента.

filters

Устанавливает коллекцию фильтров элементу формы. Если объект не является элементом формы, будет брошено исключение.

validators

Устанавливает коллекцию валидаторов элементу формы. Если объект не является элементом формы, будет брошено исключение.

required

Устанавливает атрибут required элементу формы и добавляет элементу валидатор umi\validation\toolbox\IValidatorTools::TYPE_REQUIRED. Если объект не является элементом формы, будет брошено исключение.

Использование формы

Основной задачей формы является преобразование значений в заданный формате. Т.е. на вход форме даются значения, пришедшие от пользователя, а на выход форма возвращает значения в заданном формате.

Формы создаются с помощью метода компонента umi\form. Рассмотрим форму, созданную на основе нашей конфигурации.

$form = $entityFactory->createForm(
    require __DIR__ . '/fixture/form.php'
);

$form->setData(
    [
        'login'    => 'Umi',
        'password' => 'Umi3',
        'submit'   => 'Login',
    ]
);

var_dump($form->isValid());
var_dump($form->getData());

В результате выполнения данной программы мы получим:

bool(true)
array(2) {
  'login' =>
  string(3) 'Umi';
  'password' =>
  string(4) 'Umi3'
}

Видно, что данные были преобразованы к удобному для использования виду. К примеру, значение кнопки отправки было удалено из итогового результата.

Также значения этого массива будут отфильтрованы и проверены валидаторами. Если какое либо значение из пришедших не пройдет валидацию, то вся форма будет считаться неверной, но значения в ней по прежнему будут доступны.

Вывод формы

Форма является одним из элементов взаимодействия с пользователем. По этой причине необходимо иметь возможность вывода формы на экран пользователя. Для этого у групп элементов, коллекций и форм есть возможность доступа к определенным элементам, атрибутам или опциям, а также их последовательный перебор. Для более удобного вывода элементов формы в системе существует View Helper для отображения элементов формы.

Рассмотрим пример вывода формы на экран пользователя, используя шаблонизатор PHP:

<?= $this->form()->openTag($form) ?>
<h1>Sample form</h1>
<?php foreach ($form as $element): ?>
<label>
    <?= $element->getLabel() ?>:
    <?= $this->form()->formElement($element); ?>
</label>
<?php endforeach; ?>
<?= $this->form()->closeTag(); ?>

Элементы формы

Поле формы

umi\form\element\IElement является простым полем формы. В компоненте umi\form присутствует большой набор полей формы:

umi\form\element\Button::TYPE_NAME

Поле формы - кнопка. Значением данного поля является его метка(заголовок), по этому значение данного поля не может быть изменено. По умолчанию значение этого элемента не попадает в результирующие данные формы.

umi\form\element\Checkbox::TYPE_NAME

Поле формы - флажок. Значение по умолчанию данного поля 1. Флажок может иметь 2 состояния: значение или False. При установке своего значения он автоматически изменяет значение атрибута checked.

umi\form\element\CSRF::TYPE_NAME

Скрытое поле формы защиты от CSRF уязвимостей (см. также Cross-site request forgery). Элемент этого типа проверяет установленное значение с установленным в сессии токеном. Если токены не совпадают, то форма считается не верной. Элемент сгенерирует токен в случае его отсутствия.

umi\form\element\Hidden::TYPE_NAME

Скрытое поле формы.

umi\form\element\MultiCheckbox::TYPE_NAME

Поле формы - флажки.

umi\form\element\Password::TYPE_NAME

Поле ввода пароля формы.

umi\form\element\Radio::TYPE_NAME

Поле формы - переключатель.

umi\form\element\Reset::TYPE_NAME

Поле формы - кнопка сброса. Реализует такие же свойства, как и кнопка.

umi\form\element\Select::TYPE_NAME

Поле формы - поле выбора.

umi\form\element\Submit::TYPE_NAME

Поле формы - кнопка отправки формы. Реализует такие же свойства, как и кнопка.

umi\form\element\Text::TYPE_NAME

Текстовое поле формы.

umi\form\element\Textarea::TYPE_NAME

Многострочное текстовое поле формы.

umi\form\element\Color::TYPE_NAME

HTML5 поле формы - выбор цвета.

umi\form\element\Date::TYPE_NAME

HTML5 поле ввода даты.

umi\form\element\DateTime::TYPE_NAME

HTML5 поле ввода даты и времени.

umi\form\element\Email::TYPE_NAME

HTML5 поле ввода Email адреса.

umi\form\element\Month::TYPE_NAME

HTML5 поле выбора месяца.

umi\form\element\Number::TYPE_NAME

HTML5 поле ввода числа.

umi\form\element\Phone::TYPE_NAME

HTML5 поле ввода телефонного номера.

umi\form\element\Range::TYPE_NAME

HTML5 поле выбора диапазона значений.

umi\form\element\Search::TYPE_NAME

HTML5 поле для ввода поискового запроса.

umi\form\element\Time::TYPE_NAME

HTML5 поле ввода времени.

umi\form\element\Url::TYPE_NAME

HTML5 поле ввода URL адреса.

umi\form\element\Week::TYPE_NAME

HTML5 поле ввода недели.

Группа элементов

Группы элементов удобно использовать для визуального разделения элементов. Рассмотрим форму с группой полей:

return [
    'name'     => 'sampleRegisterForm',
    'action'   => '/register',
    'method'   => 'post',
    'elements' => [
        'identity' => [
            'type'     => Fieldset::TYPE_NAME,
            'elements' => [
                'login'    => [
                    'type'  => Text::TYPE_NAME,
                    'label' => 'Login'
                ],
                'password' => [
                    'type'  => Password::TYPE_NAME,
                    'label' => 'Password'
                ],
            ]
        ],
        'personal' => [
            'type'     => Fieldset::TYPE_NAME,
            'elements' => [
                'firstName' => [
                    'type'  => Text::TYPE_NAME,
                    'label' => 'First name'
                ],
                'lastName'  => [
                    'type'  => Text::TYPE_NAME,
                    'label' => 'Last name'
                ],
            ]
        ]
    ]
];

Использование описанной формы:

$form->setData(
    [
        'login'     => 'email',
        'password'  => '',
        'firstName' => 'Name',
        'lastName'  => 'Sname',
        'submit'    => 'Register'
    ]
);

echo $form
        ->getElement('identity')
            ->getElement('login')
                ->getValue();

Заметим, что группа полей не оказывает влияние на именования элементов. Данные в форму приходят в таком же виде, как и при отсутствии группы элементов.

Коллекция элементов

Коллекция элементов - это группа полей, количество элементов которой задается динамически, при заполнении формы. Таким образом, коллекция элементов изменяет содержащиеся в ней элементы в момент установки в нее значения. Она также оказывает влияние на именования элементов, содержащихся в ней.

Рассмотрим пример формы, содержащей коллекцию:

return [
    'name'     => 'sampleForm',
    'action'   => '/addPost',
    'method'   => 'post',
    'elements' => [
        'name'   => [
            'type'  => Text::TYPE_NAME,
            'label' => 'Post name'
        ],
        'tag'    => [
            'type'     => Collection::TYPE_NAME,
            'elements' => [
                'firstName' => [
                    'type'  => Text::TYPE_NAME,
                    'label' => 'Tag'
                ],
            ]
        ],
        'submit' => [
            'type'  => Submit::TYPE_NAME,
            'label' => 'Add post'
        ],
    ]
];

Попробуем установить данные в такую форму. Допустим пользователь захотел добавить "New Post" с 3мя тэгами:

$form->setData(
    [
        'name'   => 'New post',
        'tag'    => [
            'Tag1',
            'Tag2',
            'Tag3'
        ],
        'submit' => 'Add post'
    ]
);

var_dump($form->getData());

$tags = iterator_to_array($form->getElement('tag'));
var_dump(count($tags));

Результатом работы программы будет:

array(2) {
  'name' => string(8) "New post"
  'tag' => array(3) {
    [0] => string(4) "Tag1"
    [1] => string(4) "Tag2"
    [2] => string(4) "Tag3"
  }
}

int(3)

В данном примере, коллекция элементов автоматически создала 3 элемента внутри себя, с соответствующими значениями.

Дочерняя форма

Дочерние формы, в отличии от групп полей, оказывают влияние на именование элементов. Таким образом, дочерние формы не используются для декорирования. Они помогают разделить данные на части.

Это может быть удобно в случае, если форма редактирует несколько объектов одновременно. В таком случае, использование дочерних форм упростит установку данных. Дочерние формы, также позволяют использовать более удобные схемы биндинга объектов в форму.

Рассмотрим пример формы с дочерней формой внутри:

return [
    'name'     => 'sampleForm',
    'elements' => [
        'name'     => [
            'type' => Text::TYPE_NAME,
        ],
        'passport' => [
            'type'     => Form::TYPE_NAME,
            'label'    => 'Passport',
            'elements' => [
                'series' => [
                    'type'  => Text::TYPE_NAME,
                    'label' => 'Series'
                ],
                'number' => [
                    'type'  => Text::TYPE_NAME,
                    'label' => 'Number'
                ]
            ]
        ]
    ]
];

Эта форма редактирования профиля пользователя, в которой находится дочерняя форма редактирования паспортных данных. Паспортные данные хранятся в системе в отдельном объекте, и их обработка должна происходить отдельно.

Рассмотрим формат возвращаемых данных формой:

$form = $entityFactory->createForm(require 'form.php');
var_dump($form->getData());;

Эта программа выведет на экран:

array(2) {
  'name' => NULL
  'passport' => array(2) {
    'series' => NULL
    'number' => NULL
  }
}

Паспортные данные в данном примере представляют из себя отдельный массив, что позволяет более удобно оперировать с ними.

Расширенное использование

Data Binding

До сих пор мы получали данные из формы в виде массива, но во многих случаях было бы удобно связать объект с формой на прямую таким образом, что изменение кого либо из них автоматически изменяло бы связанного. Для таких целей и служит связывание данных или Data Binding.

Рассмотрим пример из Главы 3.4. Данную форму удобно было бы напрямую связать с объектом, и не заботиться об обработке данных. Давайте рассмотрим классы, которые отвечают за обработку данных этой формы: Customer и Passport. Для поддержки связывания данных, они должны реализовать интерфейс umi\form\binding\IDataBinding, а также добавить события об изменении содержащихся в них данных.

Пример кода таких классов:

<?php
namespace example\form;

use umi\event\TEventObservant;
use umi\form\binding\IDataBinding;

class Customer implements IDataBinding
{
    use TEventObservant;

    protected $name;

    protected $passport;

    public function __construct()
    {
        $this->passport = new Passport();
    }

    public function getData()
    {
        return [
            'name'     => $this->name,
            'passport' => $this->passport
        ];
    }

    public function setData(array $data)
    {
        $this->name = $data['name'];
        $this->passport = $data['passport'];
    }

    public function getPassport()
    {
        return $this->passport;
    }

    public function setName($name)
    {
        $this->name = $name;
        $this->fireEvent(
            IDataBinding::EVENT_UPDATE,
            [
                'name' => $name
            ]
        );

        return $this;
    }

    public function getName()
    {
        return $this->name;
    }
}

class Passport implements IDataBinding
{

    use TEventObservant;

    protected $series;

    protected $number;

    public function getData()
    {
        return [
            'series' => $this->series,
            'number' => $this->number
        ];
    }

    public function setData(array $data)
    {
        $this->series = $data['series'];
        $this->number = $data['number'];
    }

    public function setNumber($number)
    {
        $this->number = $number;
        $this->fireEvent(
            IDataBinding::EVENT_UPDATE,
            [
                'number' => $number
            ]
        );

        return $this;
    }

    public function getNumber()
    {
        return $this->number;
    }

    public function setSeries($series)
    {
        $this->series = $series;
        $this->fireEvent(
            IDataBinding::EVENT_UPDATE,
            [
                'series' => $series
            ]
        );

        return $this;
    }

    public function getSeries()
    {
        return $this->series;
    }

}

Установим для формы связывание с объектом класса Customer.

$form->bindObject($customer);

В результате выполнения этого кода, значения объекта и формы будут связаны. Также, будет установлена связь объекта класса Passport и дочерней формы passport.

Предупреждение

В результате связывания значений формы с объектом, метод umi\form\IForm::getData() будет возвращать объект.

Рассмотрим установку данных в такую форму:

$form->setData([
		'name' => 'My name',
		'passport' => [
			'series' => '0000',
			'number' => '123456',
		]
	]);

var_dump($customer->getName()); // выводит на экран string(7) "My name"

$customer->getPassport()->setNumber('654321');
$number = $form->getElement('passport')->getElement('number');
var_dump($number->getValue()); // выводит на экран string(6) "654321"

Фильтры и валидаторы

Поля формы могут содержать набор фильтров и валидаторов. В таком случае значение, которое устанавливается в элемент, сначала проходит фильтрацию, а после валидацию. Отфильтрованные данные всегда доступны в элементе, даже если валидация не была успешной.

Рассмотри форму авторизации из Главы 2. Добавим к форме несколько фильтров и валидаторов с помощью аннотаций в конфигурации формы:

	return [
		'name' => 'sampleForm',
		'action' => '/sample',
		'method' => 'post',

		'elements' => [
			'login' => [
				'type' => Text::TYPE_NAME,
				'label' => 'Login',
				'filters' => [
					IFilterTools::TYPE_STRING_TRIM => []
				],
				'validators' => [
					IValidatorTools::TYPE_REQUIRED => []
				]
			],
			'password' => [
				'type' => Password::TYPE_NAME,
				'label' => 'Password'
			],
			'submit' => [
				'type' => Submit::TYPE_NAME,
				'label' => 'Login'
			],
		]
	];

Теперь, поле login формы является обязательным и пробелы по краям строки будут обрезаны. Попробуем установить в форму значения:

	$form->setData([
		'login' => '  Umi  ',
		'password' => 'Umi3',
		'submit' => 'Login',
	]);

	var_dump($form->isValid());
	var_dump($form->getData());

Данная программа выведет на экран:

bool(true)
array(2) {
  'login' => string(3) "Umi"
  'password' => string(4) "Umi3"
}

Рассмотрим пример установки неверных данных:

	$form->setData([
		'login' => '      ',
		'password' => 'Umi3',
		'submit' => 'Login',
	]);

	var_dump($form->isValid());
	var_dump($form->getData());

В результате на экран будет выведена следующая информация:

bool(false)
array(2) {
  'login' => NULL
  'password' => string(4) "Umi3"
}

Также, у коллекцию фильтров и валидаторов для каждого поля можно изменить в процессе выполнения программы, например дополнив ее несколькими валидаторами. Для этого у полей формы существуют методы: umi\form\element\IElement::getFilters() и umi\form\element\IElement::getValidators(), возвращающие коллекции фильтров и валидаторов соответственно.
