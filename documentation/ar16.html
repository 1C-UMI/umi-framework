umi\orm

Введение

Компонент umi\orm реализует технологию Object Relational Mapping (Объектно-реляционное отображение), позволяя представить запись из базы данных в виде экземпляра доменного объекта (паттерн Data Mapper).

По умолчанию, в процессе работы приложения, все объекты загружаются единожды, при необходимости только догружая свои свойства. При обращении к объекту, объект ищется в карте соответствия (паттерн Identity Map). Карта соответствия очищается в момент завершения работы приложения, но существует возможность очистить ее принудительно.

Компонент umi\orm позволяет выполнять такие действия с объектами как добавление, чтение, обновление и удаление (CRUD) без знаний о том, какой конкретный драйвер баз данных используется.

Чтобы использовать umi\orm достаточно подключить его toolbox и сконфигурировать списки необходимых для работы коллекций объектов и их метаданные.

Важно

Компонент umi\orm не может работать без использования компонента для работы с базами данных umi\db, так как использует его API.

Базовое использование

Подключение

Для того чтобы использовать компонент umi\orm в приложении, необходимо зарегестрировать его toolbox в umi\toolkit. В приложении, основанном на скелетон приложение, для этого достаточно раcширить описание Bootstrap::OPTION_TOOLKIT в файле /configuration/bootstrap.php подключением файла с конфигурацией инструментов компонентa umi\orm:

/**
* Используемый компонент.
*/
Bootstrap::OPTION_TOOLKIT => [
    ...,
    require($libraryPath . '/orm/toolbox/config.php') //подключаем ORM
]

В стандартной конфигурации компонет umi\orm предоставляет следующие сервисы:

  • umi\orm\collection\ICollectionManager - сервис для работы с коллекциями объектов

  • umi\orm\manager\IObjectManager - сервис для работы с объектами

  • umi\orm\metadata\IMetadataManager - сервис для работы с метаданными коллекций объектов

  • umi\orm\persister\IObjectPersister - сервис для сохранения состояния объектов в базу данных

В стандартной конфигурации компонет umi\orm предоставляет следующие aware interfaces:

  • umi\orm\collection\ICollectionManagerAware - интерфейс для внедрения сервиса umi\orm\collection\ICollectionManager

  • umi\orm\manager\IObjectManagerAware - интерфейс для внедрения сервиса umi\orm\manager\IObjectManager

  • umi\orm\metadata\IMetadataManagerAware - интерфейс для внедрения сервиса umi\orm\metadata\IMetadataManager

  • umi\orm\persister\IObjectPersisterAware - интерфейс для внедрения сервиса umi\orm\persister\IObjectPersister

Конфигурирование

Конфигурирование компонента umi\orm для приложения, основанного на скелетон приложении, осуществляется в том же файле /configuration/bootstrap.php, что и подключение. Для этого расширяются настройки инструментов в описании Bootstrap::OPTION_SETTINGS:

/**
* Настройки компонентов.
*/
Bootstrap::OPTION_SETTINGS => [
    ...,
    
    /**
    * Настройки компонента ORM.
    */
    ORMTools::NAME => [
        'collections' => [ ... ]
        'metadata' => [ ... ],
        ...
    ]
]

Самыми важными настройками являются конфигурация используемых в приложении коллекций (collections) и конфигурация метаданных коллекций (metadata). Возможности этих конфигураций подробно описаны в секциях "Конфигурация списков коллекций" и "Конфигурация метаданных коллекций" соответственно.

Примеры использования

Чаще всего umi\orm будет использоваться в моделях и контроллерах вашего приложения, поэтому примеры использования можно посмотреть в Get Started.

Коллекции объектов

umi\orm\collection\ICollection - коллекция объектов одной предметной области. Коллекция предоставляет API для простейших действий с объектами - создание, чтение и удаление. Любая коллекция описывается с помощью используемых в коллекции типов и полей, на основе которых формируются свойства объектов, и информации о связанной таблице базы данных для маппинга данных.

Типы коллекций

Простая коллекция объектов

umi\orm\collection\ISimpleCollection - коллекция простых доменных объектов.

Простая коллекция иерархических объектов

umi\orm\collection\IHierarchicCollection - коллекция иерархических объектов, имеющих родственные связи.

Коллекция связанных иерархических объектов

umi\orm\collection\ILinkedHierarchicCollection - особая коллекция иерархических объектов, имеющих родственные связи, связанная с общей иерахической коллекцией.

Общая иерархия разнотипных объектов

umi\orm\collection\ICommonHierarchy - особая коллекция, не имеющая собственных объектов. Такая коллекция объединяет иерархически связанные между собой объекты из разных коллекций связанных иерархических объектов (umi\orm\collection\ILinkedHierarchicCollection). Общая иерархия предназначена для построения общих деревьев разнотипных объектов. Наиболее наглядный пример - построение общей структуры сайта, имеющего страницы из разных коллекций.

Общая иерархия является поставщиком и генератором идентификаторов для объектов связанных с ней иерархических коллекций. Помимо того она может содержать любой возможный набор общих свойств связанных с ней иерархических коллекций, что обеспечивает быструю иерархическую выборку разнотипных объектов. При изменении и создании объектов в связанных коллекциях, значения общих свойств дублируются в иерархию автоматически.

Выборки объектов из общей иерархии представляют собой набор объектов с частично загруженными (общими) свойствами. Каждый объект принадлежит собственной иерархической коллекции, при попытке обратиться к уникальным свойствам объекта, он будет догружен из соответсвующей коллекции.

Конфигурация списков коллекций

Список используемых коллекций задается в конфигурации "Менеджера коллекций" (umi\orm\collection\ICollectionManager), одного из сервисов компонента umi\orm, обеспечивающего доступ к коллекциям.

Конфигурация любой коллекции обязательно должна содержать опцию type, указывающую на тип коллекции. Возможные варианты типов:

  • umi\orm\collection\ICollectionFactory::TYPE_SIMPLE - простая коллекция объектов

  • umi\orm\collection\ICollectionFactory::TYPE_SIMPLE_HIERARCHIC - простая коллекция иерархических объектов

  • umi\orm\collection\ICollectionFactory::TYPE_LINKED_HIERARCHIC - коллекция связанных иерархических объектов

  • umi\orm\collection\ICollectionFactory::TYPE_COMMON_HIERARCHY - общая иерархия иерархически связанных разнотипных объектов

Конфигурация коллекции связанных иерархических объектов обязательно содержит опцию hierarchy - ссылку на общую иерархию.

Конфигурация любой коллекции может содержать опцию class - полное имя класса коллекции, реализующего бизнес-логику конкретной коллекции объектов.

Пример конфигурации списка коллекций: 

'collections' => [
	'system_hierarchy' => [
		'type' => ICollectionFactory::TYPE_COMMON_HIERARCHY
	],
	'blogs_post' => [
		'type' => ICollectionFactory::TYPE_LINKED_HIERARCHIC,
		'class' => 'vendor\collection\blog\PostCollection',
		'hierarchy' => 'system_hierarchy'
	],
	'system_menu' => [
		'type' => ICollectionFactory::TYPE_SIMPLE_HIERARCHIC,
		'class' => 'vendor\collection\menu\MenuCollection',
	],
	'users_user' => [
		'type' => ICollectionFactory::TYPE_SIMPLE,
		'class' => 'vendor\collection\user\UserCollection',
	],
		'files_file' => [
		'type' => ICollectionFactory::TYPE_SIMPLE
	]
]

Конфигурация метаданных коллекций

Метаданные коллекции описывают типы объектов коллекции, поля типов и содержат информацию о таблице в базе данных для связывания объектной и реляционной моделей. Конфигурация метаданных задается в конфигурации "Менеджера метаданных" (umi\orm\metadata\IMetadataManager), одного из сервисов компонента umi\orm, обеспечивающего доступ к метаданным коллекций.

Пример конфигурации списка метаданных для коллекций:

'metadata' => [
	'system_hierarchy' => [...],
	'blogs_post' => [...],
	'system_menu' => [...],
	'users_user' => [...],
	'files_file' => [...]
]

Метаданные коллекции

Конфигурация метаданных обязательно должна содержать следующие опции:

  • dataSource - содержит информацию об источнике данных (о таблице в базе данных) коллекции. Эта часть конфигурации имеет обязательную вложенную опцию sourceName, содержащую имя таблицы базы данных, а также 2 необязательных опции masterServerId и slaveServerId. Опции masterServerId и slaveServerId содержат в себе ссылки на используемые сервера баз данных, описанные в конфигурации компонента для работы с базами данных umi\db. Если эти опции не указаны, то считается, что источник данных находится на дефолтных серверах.

  • fields - список конфигураций всех полей коллекции на всю иерархию типов коллекции, на основе которых будут формированы свойства объектов. Возможные конфигурации отдельных полей описаны в секции "Свойства объекта".

  • types - список конфигураций иерархии типов объектов. Конфигурация каждого типа обязательно содержит опцию fields со списком полей, необходимым для создания свойств, которыми должен обладать объект данного типа. Необязательная опция objectClass содержит полное имя класса создаваемых объектов данного типа.

Каждая коллекция обязательно имеет поля "Идентификатор" (id), "GUID" (guid), "Тип объекта" (type) и "Версия объекта" (version). Каждая иерархическая коллекция в дополнение обязательно имеет поля "Родитель" (parent), "Материализованный путь" (mpath), "Последнюю часть ЧПУ" (slug), "URI" (uri), "Количество дочерних элементов" (childCount), "Порядок следования в иерархии" (order) и "Уровень вложенности в иерархии" (level).

Каждая коллекция обязательно имеет базовый тип объектов (umi\orm\metadata\IObjectType::BASE). Вся иерархия типов коллекции строится от базового типа. Иерархическая связанность типов отражается в их именах, полное имя типа состоит из полного имени его родителя и собственного имени, разделенных сепаратором umi\orm\metadata\IObjectType::NAME_SEPARATOR. Это правило не относится к дочерним типам базового типа. Типы не могут отрицать поведение своих родителей, поэтому каждый дочерний тип должен содержать все свойства, которые есть у его родителя.

Рассмотрим пример возможной конфигурации метаданных простой коллекции объектов "Файл". Т.к. файлы разных типов обладают разными характеристиками, выделим несколько типов объектов - текстовый файл (text), изображение (image) и медиа-файл (media), который можно разделить на музыкальный файл (media.music) и видео-файл (media.video).

Объекты

Доменный объект

umi\orm\object\IObject - объект бизнес-логики (напр., пользователь, товар в каталоге, пост блога). Каждый объект характеризуется принадлежностью к определенной коллекции, конкретным типом, заданным в рамках коллекции, и набором свойств, определенным через поля типа.

Иерархический доменный объект

umi\orm\object\IHierarchicObject - доменный объект, имеющий родственные связи. Это объект из простой иерархической коллекции umi\orm\collection\ISimpleHierarchicCollection или из коллекции связанных иерархических объектов umi\orm\collection\ILinkedHierarchicCollection.

Свойства объектов

Каждый доменный объект в umi\orm характеризуется набором свойств, заданным через поля его типа в метаданных коллекции объекта. Объект может иметь простые скалярные свойства, локализуемые скалярные свойства, свойства для поддержки связей между объектами и вычисляемые свойства.

Доступ к значению свойства объекта можно получить несколькими способами:

  • через определенный для свойства метод доступа, задаваемый в конфигурации соответствующего поля через опцию accessor ($user->getLogin()),

  • через обращение к свойству как к публичному свойству объекта ($user->login),

  • через обращение к объекту как к массиву ($user['login']),

  • через метод umi\orm\object\IObject::getValue($propName, $localeId = null) ($user->getValue('login')).

При обращении к значению свойства любым из способов будет вызван либо заданный accessor, либо дефолтный метод доступа getValue(), если accessor не задан.

Изменить значение свойства можно несколькими способами:

  • через определенный для свойства метод изменения значения, задаваемый в конфигурации соответствующего поля через опцию mutator ($user->setLogin($login)),

  • через обращение к свойству как к публичному свойству объекта ($user->login = $login),

  • через обращение к объекту как к массиву ($user['login'] = $login),

  • через метод umi\orm\object\IObject::setValue($propName, $value, $localeId = null) ($user->setValue('login', $login)).

Если для изменения значения свойства задана опция mutator, то при изменении значения свойства любым из способов управление передается методу указонному в этой опции.

Наличие опций mutator и accessor у поля подразумевает, что для типа объекта, использующего это поле, в конфигурации задана опция objectClass, содержащая полное имя класса создаваемых объектов. Этот класс должен содержать реализацию методов указанных в опциях mutator и accessor.

Конфигурация любого поля, формирующего свойство, обязательно должна содержать следующие опции:

  • type - тип поля, соответствует значению одной из констант umi\orm\metadata\field\IField

Конфигурация любого свойства может содержать следующие опции:

  • accessor - метод доступа к значению свойства

  • mutator - метод измения значения свойства

  • readOnly - признак того, что свойство доступно только для чтения

Пример конфигурации свойства "Версия объекта" (version), которое должно присутствовать в любой коллекции: 

'version' => ['type' => IField::TYPE_VERSION, 'columnName' => 'version', 'accessor' => 'getVersion', 'mutator' => 'setVersion', 'defaultValue' => 1]

Поддерживаемые фреймворком типы полей

Числовые типы полей:

  • umi\orm\metadata\field\IField::TYPE_BOOL - булевый тип

  • umi\orm\metadata\field\IField::TYPE_DECIMAL - числовой тип с фиксированной точностью и масштабом

  • umi\orm\metadata\field\IField::TYPE_INTEGER - целочисленный тип

  • umi\orm\metadata\field\IField::TYPE_REAL - числовой тип с плавающей запятой

Строковые типы полей, поддерживаемые фреймворком:

  • umi\orm\metadata\field\IField::TYPE_BLOB - тип двоичных данных

  • umi\orm\metadata\field\IField::TYPE_CHAR - строковый тип фиксированной длины

  • umi\orm\metadata\field\IField::TYPE_STRING - строковый тип переменной длины

  • umi\orm\metadata\field\IField::TYPE_TEXT - текстовый тип

Типы полей для работы с временем и датами:

  • umi\orm\metadata\field\IField::TYPE_DATE - тип даты

  • umi\orm\metadata\field\IField::TYPE_DATE_TIME - тип даты вместе с указанием времени

  • umi\orm\metadata\field\IField::TYPE_TIME - тип времени

  • umi\orm\metadata\field\IField::TYPE_TIMESTAMP - тип POSIX-времени

Специальные типы полей:

  • umi\orm\metadata\field\IField::TYPE_COUNTER - тип-счетчик

  • umi\orm\metadata\field\IField::TYPE_GUID - тип глобального идентификатора объекта (GUID)

  • umi\orm\metadata\field\IField::TYPE_IDENTIFY - тип основного идентификатора объекта (первичный ключ таблицы коллекции)

  • umi\orm\metadata\field\IField::TYPE_LEVEL - тип иерархического уровня вложенности

  • umi\orm\metadata\field\IField::TYPE_MPATH - тип материализованного пути объекта

  • umi\orm\metadata\field\IField::TYPE_ORDER - тип порядка в иерархии

  • umi\orm\metadata\field\IField::TYPE_SLUG - тип последней части ЧПУ

  • umi\orm\metadata\field\IField::TYPE_URI - тип URI

  • umi\orm\metadata\field\IField::TYPE_VERSION - тип версии объекта

Типы полей, реализующих связи между объектами:

  • umi\orm\metadata\field\IField::TYPE_BELONGS_TO - тип многие к одному

  • umi\orm\metadata\field\IField::TYPE_HAS_MANY - тип один ко многим

  • umi\orm\metadata\field\IField::TYPE_HAS_ONE - тип один к одному

  • umi\orm\metadata\field\IField::TYPE_MANY_TO_MANY - тип многие ко многим

Расширить список возможных типов полей можно через конфигурацию "Фабрики метаданных" (umi\orm\metadata\IMetadataFactory), одного из сервисов umi\orm, обеспечивающего создание всех сущностей метаданных.

Простые скалярные свойства

Cкалярные свойства - это свойства, имеющие скалярные значения одного из PHP-типов (boolean, integer, string). Подразумевается, что простые скалярные свойства не имеют локализации, и их значения не зависят от других свойств объекта.

Конфигурация поля для простого скалярного свойства обязательно должна содержать следующие опции:

  • columnName - имя столбца в таблице базы данных для хранения значение свойства, имя столбца может отличаться от имени поля

Конфигурация поля для простого скалярного свойства может содержать следующие опции:

  • defaultValue - значение свойства по умолчанию

Локализуемые скалярные свойства

Локализуемые свойства - это свойства, значениях которых имеют или могут иметь перевод для каждой используемой локали. Значения локализуемого свойства всегда зависит от установленной текущей локали. Если перевод для текущей локали не задан, значением свойства будет значение для локали по умолчанию.

Конфигурация поля для локализуемого свойства помимо стандартных опций поля содержит следующие опции:

  • columnName - имя столбца в таблице базы данных для хранения значение свойства в дефолной локали

  • defaultValue - значение свойства по умолчанию в дефолтной локали (необязательная опция)

  • localizations - список локализаций поля, описывающий имена столбцов и дефолтные значение для каждой локали, включая дефолтную.

Пример конфигурации поля для локализуемого свойства "Заголовок" (title) без значений по умолчанию: 

'title' => [
                'type' => IField::TYPE_STRING,
                'columnName' => 'title',
                'localizations' => [
				    'ru-RU' => ['columnName' => 'title'],
				    'en-US' => ['columnName' => 'title_en']
			     ],
			     'accessor' => 'getTitle',
			     'mutator' => 'setTitle'
		   ]

Пример конфигурации поля для локализуемого свойства "Заголовок" (title) со значениями по умолчанию: 

'title' => [
                'type' => IField::TYPE_STRING,
                'columnName' => 'title',
                'defaultValue' => 'Новая страница',
                'localizations' => [
				    'ru-RU' => ['columnName' => 'title', , 'defaultValue' => 'Новая страница'],
				    'en-US' => ['columnName' => 'title_en', 'defaultValue' => 'New page']
			     ],
			     'accessor' => 'getTitle',
			     'mutator' => 'setTitle'
		   ]

Связи между объектами

Компонент umi\orm предлагает 4 типа полей, для реализации связей между объектами.

umi\orm\metadata\field\relation\BelongsToRelationField

На основе поля umi\orm\metadata\field\relation\BelongsToRelationField реализуется связь "Многие к одному". Такая связь возникает, когда нескольким объектам одной коллекции соответствует один объект из другой или той же коллекции. Например, несколько позиций заказа входят в состав одного заказа интернет-магазина, или несколько страниц контента являются дочерними по отношению к общему родителю. Таким образом, можно сказать, что позиция заказа принадлежит заказу, а дочерняя страница принадлежит родительской. Значением свойства, основанного на данном поле, будет объект связанной коллекции (umi\orm\object\IObject).

Конфигурация поля, реализуещего связь "Многие к одному" содержит следующие опции:

  • type - тип поля, соответствует значению константы umi\orm\metadata\field\IField::TYPE_BELONGS_TO,

  • columnName - имя столбца в таблице базы данных для хранения значение свойства,

  • target - имя коллекции, с которой осуществляется связь.

Пример конфигурации поля "Заказ" (order) в коллекции "Позиции заказа" (emarket_order_item) для связи с коллекцией "Заказ в магазине" (emarket_order): 

'order' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'order_id', 'target' => 'emarket_order']

Пример конфигурации поля "Родительская страница" (parent) в коллекции "Страница контента" (content_page) для связи с той же самой коллекцией: 

'parent' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'pid', 'target' => 'content_page']

Важно

Поля, реализующие связь "Многие к одному" являются хранителями связи по отношению ко всем остальным типам связи, так как значения соответствующих свойств хранятся в базе данных только на стороне этой связи.

umi\orm\metadata\field\relation\HasManyRelationField

На основе поля umi\orm\metadata\field\relation\HasManyRelationField реализуется связь "Один ко многим". Такая связь возникает, когда одному объекту может соответствовать несколько принадлежащих только ему объектов той же или другой коллекции. Например, страница контента может иметь несколько дочерних страниц, а заказ в интернет-магазине имеет несколько позиций. Значением свойства, основанного на данном поле, будет набор объектов из связанной коллекции (umi\orm\objectset\IObjectSet). Задать значение такому свойству напрямую невозможно, для этого используется связанное свойство с типом связи "Многие к одному".

Т.к. значение связи "Один ко многим" хранится на стороне связи "Многие к одному", конфигурация поля не имеет опции columnName, а имеет опцию-ссылку на связанное свойство (targetField) и также содержит ссылку на связанную коллекцию (target), которой принадлежат объекты-значения.

Конфигурация поля, реализующего связь "Один ко многим" содержит следующие опции:

  • type - тип поля, соответствует значению константы umi\orm\metadata\field\IField::TYPE_HAS_MANY,

  • target - имя коллекции, с которой осуществляется связь,

  • targetField - имя поля типа umi\orm\metadata\field\IField::TYPE_BELONGS_TO в связанной коллекции, ссылающегося на коллекцию для которой формируется связь.

Пример конфигурации поля "Дочернии страницы" (children) в коллекции "Страница контента" (content_page) для связи с той же самой коллекцией: 

'children' => ['type' => IField::TYPE_HAS_MANY, 'target' => 'content_page', 'targetField' => 'parent', 'readOnly' => true]

Конфигурация связанного поля "Родительская страница" (parent) в коллекции "Страница контента" (content_page) со связью "Многие к одному": 

'parent' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'pid', 'target' => 'content_page']

Пример конфигурации поля "Позиции заказа" (items) в коллекции "Заказ в магазине" (emarket_order) для связи с коллекцией "Позиции заказа" (emarket_order_item): 

'items' => ['type' => IField::TYPE_HAS_MANY, 'target' => 'emarket_order_item', 'targetField' => 'order', 'readOnly' => true]

Конфигурация связанного поля поля "Заказ" (order) в коллекции "Позиции заказа" (emarket_order_item) со связью "Многие к одному": 

'order' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'order_id', 'target' => 'emarket_order']

umi\orm\metadata\field\relation\HasOneRelationField

На основе поля umi\orm\metadata\field\relation\HasOneRelationField реализуется связь "Один к одному". Такая связь возникает, когда для одного объекта имеется только один принадлежащий только ему объект из другой или той же коллекции. Связь "Один к одному" является частным случаем связи "Один ко многим". Например, у покупателя интернет-магазина может быть одно юридическое лицо, и это юридическое лицо не может принадлежать другому покупателю. Значением соответствующего свойства является umi\orm\object\IObject из связанной коллекции. Такая связь является равнозначной для обеих связанных коллекций, поэтому хранителем связи может быть любая из двух коллекций.

Рассмотрим, пример конфигурации, если хранителем связи является коллекция юридических лиц:

Конфигурация поля "Юридическое лицо" (legalPerson) в коллекции "Покупатель" (customers_customer): 

'legalPerson' => ['type' => IField::TYPE_HAS_ONE, 'target' => 'customers_legal_person', 'targetField' => 'customer', 'readOnly' => true]

Конфигурация связанного поля "Покупатель" (customer) в коллекции "Юридическое лицо" (customers_legal_person) со связью "Многие к одному" : 

'customer' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'customer_id', 'target' => 'customers_customer']

umi\orm\metadata\field\relation\ManyToManyRelationField

На основе поля umi\orm\metadata\field\relation\ManyToManyRelationField реализуется связь "Многие ко многим". Этот тип связи используется, когда нескольким объектам одной коллекции соответствуют несколько объектов другой коллекции. Примером такой связи является ситуация, когда у каждого поста в блоге есть свой список тегов, при этом один и тот же тег может соответствовать нескольким постам одновременно.

Связь "Многие ко многим" всегда является зеркальной, для ее построения используется вспомогательная коллекция-мост с двумя соответствующими связями "Многие к одному". Значением свойства на основе поля со связью "Многие ко многим" является umi\orm\objectset\IManyToManyObjectSet - набор объектов из связанной коллекции. Изменить значение свойства можно только добавляя и удаляя объекты в результирующем наборе значений, напрямую значение выставить нельзя. Также через набор объектов можно редактировать свойства связи - объекта из коллекции-моста. Так, допустим, для списка тегов поста это может быть порядок следования тегов в списке.

Конфигурация связи "Многие ко многим" содержит следующие опции:

  • type - тип поля, соответствует значению константы umi\orm\metadata\field\IField::TYPE_MANY_TO_MANY,

  • target - имя коллекции, с которой осуществляется связь,

  • bridge - имя коллекции-моста, через которую осуществляется связь,

  • targetField - имя поля типа umi\orm\metadata\field\IField::TYPE_BELONGS_TO в коллекции-мосте, ссылающегося на связанную коллекцию,

  • relatedField - имя поля типа umi\orm\metadata\field\IField::TYPE_BELONGS_TO в коллекции-мосте, ссылающегося на коллекцию, для которой формируется связь.

Конфигурация поля "Посты" (posts) в коллекции "Тег" (blogs_tag): 

'posts' => ['type' => IField::TYPE_MANY_TO_MANY, 'target' => 'blogs_post', 'bridge' => 'blogs_post_tag', 'relatedField' => 'tag', 'targetField' => 'post', 'readOnly' => true]

Конфигурация поля "Теги" (tags) в коллекции "Пост" (blogs_post): 

'tags' => ['type' => IField::TYPE_MANY_TO_MANY, 'target' => 'blogs_tag', 'bridge' => 'blogs_post_tag', 'relatedField' => 'post, 'targetField' => 'tag', 'readOnly' => true]

Конфигурация поля "Тег" (tag) и поля "Блог" (blog) в коллекции-мосте "Теги постов" (blogs_post_tag): 

'tag' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'tag_id', 'target' => 'blogs_tag'],
'post' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'post_id', 'target' => 'blogs_post']

Вычисляемые свойства

Вычисляемые свойства - это свойства, значения для которых формируются в зависимости от конкретных условий. Такие свойства удобно использовать для денормализации базы данных. Напрмер, чтобы получить количество дочерних страниц у страницы контента, гораздо быстрее получить доступ к заранее посчитанному свойству "Количество детей", чем каждый раз выполнять дополнительный запрос к базе данных для подсчета.

Базовые действия над объектами: операции CRUD

Сами объекты ничего не знают о постоянном хранилище, не умеют загружать и сохранять значения измененных свойств. Сохранение измененного состояния объектов обеспечивает синхронизатор объектов (umi\orm\persister\IObjectPersister). Синхронизатор объектов отслеживает изменяемые в бизнес-транзакции (бизнес-действии) объекты и управляет записью изменений и разрешением проблем конкуренции данных, реализуя паттерн Unit of Work. За одну бизнес-транзакцию можно одновременно добавить, изменить и удалить неограниченное количество объектов. Такой подход гарантирует то, что при невозможности по каким-то причинам сохранить любой из объектов транзакции связанные данные также не будут сохранены.

При сохранении объектов проверяется актуальность их версии (паттерн Optimistic Offline Lock ). Например, если два человека одновременно начали редактировать пост блога и создавать для него новые теги, но первый сохранил свои изменения раньше, второй не сможет сохранить свои изменения, и его теги не будут добавлены в базу данных.

Инициаторами добавления, чтения и удаления объектов выступают коллекции объектов.

Рассмотрим операции над объектами на примере работы с блогами. Допустим нам нужно добавить новый пост блога, добавить несколько новых тегов для него, отредактировать ранее неверно созданный тег и удалить ненужный: 

//получаем объект из коллекции
$unusedTag = $tagCollection->get($unusedTagGUID);
//помечаем объект на удаление
$tagCollection->delete($unusedTag);

//получаем объект из коллекции
$wrongTag = $tagCollection->get($wrongTagGUID);
//изменяем значение свойства объекта
$wrongTag->setName('newTagName');

//добавление и изменение объектов
$firstNewTag = $tagCollection->add(IObjectType::BASE)->setName('firstTag');          
$secondNewTag = $tagCollection->add(IObjectType::BASE)->setName('secondTag');

$post = $postCollection->add(IObjectType::BASE)
      ->setName('newPost')
      ->setContent('content for new post');
      
$postTags = $post->getTags();
$postTags->attach($secondNewTag)->setOrder(2);
$postTags->attach($firstNewTag)->setOrder(1);

//сохранение всех новых, модифицированных и удаленных объектов в бд, выполнение всех sql-запросов
$objectPersister->commit();

Расширенные действия над объектами

Выборки объектов

Выборки объектов всегда выполняются строго из одной коллекции с помощью селектора коллекции umi\orm\selctor\ISelector. Результатом работы селектора является итерируемый набор объектов одной коллекции - umi\orm\objectset\IObjectSet. Исключением являются выборки из общих иерархических коллекций. Объекты загружаются из базы данных в набор объектов лениво, по мере итерации результата выборки. Объекты могут быть загружены полностью или частично. Рассмотрим несколько примеров выборок.

Простая выборка объектов с уловиями

Рассмотрим возможности простой выборки с условиями по простым скалярным свойствам на примере выборки из коллекции "Товар" (catalog_product), в которой есть один базовый тип и поля "Цена" (price), "Количество" (quantity) и "Рейтинг" ('rating'):

'catalog_product' => [
	'dataSource' => [...],
	'fields' => [    
		...
        'price' => ['type' => IField::TYPE_DECIMAL, 'columnName' => 'price'],		
		'quantity' => ['type' => IField::TYPE_INTEGER, 'columnName' => 'quantity'],
		'rating' => ['type' => IField::TYPE_FLOAT, 'columnName' => 'rating'],
		...
	],
	'types' => [
	   'base' => [..., 'price', 'quantity', 'rating', ...]
	]
]

Выборка 2-х товаров, которые имеются в наличии, с ценой больше 1000 рублей и рейтингом равным 5 или ценой ниже 1000 рублей и рейтингом не ниже 4, отсортированных по рейтингу : 

$products = $productCollection->select()
    	->begin(IFieldConditionGroup::MODE_AND)
    		->where('quantity')->notEquals(0)
    		->begin(IFieldConditionGroup::MODE_OR)
    			->where('price')->more(1000)
    			->where('rating')->equals(5)
    		->end()
    		->begin(IFieldConditionGroup::MODE_OR)
    			->where('price')->less(1000)
    			->where('rating')->equalsOrMore(4)
    		->end()
    	->end()
    	->orderBy('rating', ISelector::ORDER_DESC)
    	->limit(2);
    
foreach ($products as $product) {
   ...
}

Выборка объектов с заданным типом

Рассмотрим возможности выборки с заданными типами на примере выборки из коллекции "Товар" (catalog_product), в которой есть некоторый набор полей, базовый тип, тип "Телевизор" (tv), тип "Плазменный телевизор" (tv.plazmaTv), тип "ЖК телевизор" (tv.lcdTv), тип "Фотоаппарат" (camera), тип "Цифровой фотоаппарат" (camera.digitalCamera) и тип "Пленочный фотоаппарат" (camera.filmCamera):

'catalog_product' => [
	'dataSource' => [...],
	'fields' => [...],
	'types' => [
	   'base' => [...],
	   'tv' => [...],
	   'tv.plazmaTv' => [...],
	   'tv.lcdTv' => [...],
	   'camera' => [...],
	   'camera.digitalCamera' => [...],
	   'camera.filmCamera' => [...]
	]
]

Выборка товаров с типом "Телевизор" (tv), в т.ч. всех товаров с типами дочерними типу "Телевизор" (напр., "Плазменный телевизор"), и товаров с типом "Цифровой фотоаппарат", дочерним типу "Фотоаппарат" : 

$products = $productCollection->select()
    ->types(['tv*', 'camera.digitalCamera'])
    ->getResult()
    ->fetchAll();

Выборка объектов с частичной загрузкой свойств

Рассмотрим возможности выборки с частичной загрузкой свойств объектов на примере выборки из коллекции "Товар" (catalog_product), в которой есть один базовый тип и поля "Цена" (price), "Количество" (quantity) и "Рейтинг" ('rating'):

'catalog_product' => [
	'dataSource' => [...],
	'fields' => [    
		...
        'price' => ['type' => IField::TYPE_DECIMAL, 'columnName' => 'price'],		
		'quantity' => ['type' => IField::TYPE_INTEGER, 'columnName' => 'quantity'],
		'rating' => ['type' => IField::TYPE_FLOAT, 'columnName' => 'rating'],
		...
	],
	'types' => [
	   'base' => [..., 'price', 'quantity', 'rating', ...]
	]
]

В выборке с указанием конкретных свойств для загрузки могут присутствовать только скалярные свойства.

Выборка товаров из коллекции "Товар" с частичной загрузкой только свойств "Цена" и "Количество", остальные свойства будут дозагружены после первого обращения к их значениям: 

$products = $productCollection->select()
    ->fields(['price', 'quantity'])
    ->getResult()
    ->fetchAll();

Выборка объектов с условиями по связанным объектам

Выборка объектов может быть ограничена не только значениями скалярных свойствам, но также значениями свойств свзанных объектов. Если ограничение на выборку накладывается свойствами связанных объектов, то в условии выборки указывается полный путь к свойству связанной коллекции, учитывающий названия свойств, через которые осуществляется связь.

Рассмотрим возможности выборки с условиями по связанным объектам на примере выборки из коллекции "Товар" (catalog_product), в которой есть базовый тип, тип "Платье" (dress), поле "Производитель", ссылающееся на коллекцию производителей (catalog_manufacturer), и поле "Предложения", ссылающееся на коллекцию "Товарные предложения" (catalog_product_offer). Коллекция "Товарные предложения" имеет поле "Цвет" (color), ссылающееся на коллекцию "Цвет" (color). 

'catalog_product' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'manufacturer' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'manufacturer_id', 'target' => 'catalog_manufacturer']
	   'offers' => ['type' => IField::TYPE_HAS_MANY, 'target' => 'catalog_product_offer', 'targetField' => 'product','readOnly' => true],
		...
	 ],
	'types' => [
	   'base' => [...],
	   'dress' => [..., 'manufacturer', 'offers', ...]
	]
],
'catalog_manufacturer' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'name' => ['type' => IField::TYPE_STRING, 'columnName' => 'name', 'accessor' => 'getName', 'mutator' => 'setName'],
	   'products' => ['type' => IField::TYPE_HAS_MANY, 'target' => 'catalog_product', 'targetField' => 'manufacturer, 'readOnly' => true],
		...
	 ],
	'types' => [
	   'base' => [..., 'name', 'products', ...]
	]
],
'catalog_product_offer' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'product' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'product_id', 'target' => 'catalog_product'],  		
	   'color' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'color_id', 'target' => 'color'],  		
		...
	 ],
	'types' => [
	   'base' => [..., 'product', 'color', ...]
	]
],
'color' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'name' => ['type' => IField::TYPE_STRING, 'columnName' => 'name'],
		...
	 ],
	'types' => [
	   'base' => [..., 'name', ...],
	]
]

Выборка товаров из коллекции "Товар" с типом "Платье", производителем которых является производитель с именем "Umi" (выборка по связи "Многие к одному"): 

$products = $productCollection->select()
                ->types(['dress'])
                ->where('manufacturer.name')->equals('Umi')
            ->getResult()->fetchAll();

Выборка товаров из коллекции "Товар" с типом "Платье", имеющих предложения в красном или синем цвете (выборка по связи "Один ко многим" и "Многие к одному"): 

$products = $productCollection->select()
                 ->types(['dress'])
                 ->where('offers.color.name')->in(['Красный', 'Синий'])
            ->getResult()->fetchAll();

Выборка объектов с одновременной загрузкой связанных объектов

Чтобы избежать дополнительных запросов к базе данных для получения значения свойств с типом связи "Многие к одному", можно выполнять выборку с одновременной загрузкой связанных объектов. При этом можно указать, какие конкреные свойства для связанного объекта необходимо загрузить.

Рассмотрим возможности выборки с одновременной загрузкой связанных объектов на примере выборки из коллекции "Товар" (catalog_product), в которой есть базовый тип, поле "Производитель", ссылающееся на коллекцию производителей (catalog_manufacturer), и поле "Город производства", ссылающееся на коллекцию "Город" (city). Коллекция "Город" имеет поле "Страна"(country), ссылающееся на коллекцию "Страна" (country). 

'catalog_product' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'manufacturer' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'manufacturer_id', 'target' => 'catalog_manufacturer']
	   'city' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'city_id', 'target' => 'city']  		
		...
	 ],
	'types' => [
	   'base' => [..., 'manufacturer', 'city', ...]
	]
],
'catalog_manufacturer' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'name' => ['type' => IField::TYPE_STRING, 'columnName' => 'name', 'accessor' => 'getName', 'mutator' => 'setName'],
	   'products' => ['type' => IField::TYPE_HAS_MANY, 'target' => 'catalog_product', 'targetField' => 'manufacturer', 'readOnly' => true],
		...
	 ],
	'types' => [
	   'base' => [..., 'name', 'products', ...]
	]
],
'city' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'country' => ['type' => IField::TYPE_BELONGS_TO, 'columnName' => 'country_id', 'target' => 'country'],  		
		...
	 ],
	'types' => [
	   'base' => [..., 'country', ...]
	]
],
'country' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'cities' => ['type' => IField::TYPE_HAS_MANY, 'target' => 'city', 'targetField' => 'country', 'readOnly' => true],
		...
	 ],
	'types' => [
	   'base' => [..., 'cities', ...],
	]
]

Выборка товаров из коллекции "Товар" с одновременной загрузкой производителя (manufacturer): 

$products = $productCollection->select()
                ->with('manufacturer')
            ->getResult()
            ->fetchAll;

Выборка товаров из коллекции "Товар" с одновременной загрузкой производителя только с именем и страной, где товар был произведен (связь через город производства): 

$products = $productCollection->select()
                ->with('manufacturer', ['name'])
                ->with('city.country')
            ->getResult()
            ->fetchAll;

Автоматический подбор типов в выборках

Рассмотрим возможности выборки с автоматическим подбором типов на примере выборки из коллекции "Товар" (catalog_product), в которой есть базовый тип, тип "Телевизор" (tv), тип "Плазменный телевизор" (tv.plazmaTv), тип "ЖК телевизор" (tv.lcdTv), тип "Фотоаппарат" (camera), общие поля "Цена" (price), "Название" (name) и поле "Диагональ", которое есть только у телевизоров

'catalog_product' => [
	'dataSource' => [...],
	'fields' => [
	   ...
	   'price' => ['type' => IField::TYPE_DECIMAL, 'columnName' => 'price'],		
	   'name' => ['type' => IField::TYPE_STRING, 'columnName' => 'name'],
	   'diagonal' => ['type' => IField::TYPE_FLOAT, 'columnName' => 'diagonal'],
		...
	 ],
	'types' => [
	   'base' => [..., 'price', 'name, ...],
	   'tv' => [..., 'price', 'name, ..., 'diagonal'],
	   'tv.plazmaTv' => [..., 'price', 'name, ..., 'diagonal'],
	   'tv.lcdTv' => [..., 'price', 'name, ..., 'diagonal'],
	   'camera' => [..., 'price', 'name, ...]
	]
]

Если в выборке для загрузки указаны свойства, существующие не во всех типах, или на выборку наложены ограничения по свойствам, которые есть не у всех типов, то результирующий набор объектов будет содержать только объекты подходящих типов. При загрузке товаров со свойством "Диагональ" будут выбраны только товары с типом "Телевизор" и его дочерними типами. В выборке товаров с диагональю больше 24 дюймов будут присутствовать только товары с типом "Телевизор" и его дочерними типами: 

$products = $productCollection->select()
            ->fields(['price', 'name', 'diagonal'])
        ->getResult()
        ->fetchAll();
$products = $productCollection->select()
            ->where('diagonal')->more(24)
        ->getResult()
        ->fetchAll();

Выборка объектов невозможна, если не существует ни одного типа в коллекции, в котором есть все свойства, указанные для загрузки и в условиях выборки. В таком случае будет сгенерировано исключение umi\orm\exception\InvalidArgumentException.

Выборка объектов невозможна, если хотя бы один указанный в выборке тип не содержит хотя бы одно из свойств, указанных для загрузки или в условиях выборки. В таком случае будет сгенерировано исключение umi\orm\exception\InvalidArgumentException.
